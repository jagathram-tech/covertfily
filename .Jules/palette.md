## 2024-06-12 - Icon-Only Button and Link Accessibility
**Learning:** Found multiple instances where interactive elements (such as footer social links and mobile navigation toggles) relied exclusively on FontAwesome `<i>` tags without accessible text. This causes screen readers to either ignore them or read confusing class names. Additionally, generated dynamically in JavaScript (like the remove button in `png-to-jpg.html`), the `<i>` tags still require an `aria-hidden="true"` while the parent `<button>` or `<a>` requires an `aria-label`.
**Action:** When auditing or building navigation elements with icons, I must always add a descriptive `aria-label` to the interactive wrapper (button/link) and `aria-hidden="true"` to the internal icon to prevent double-reading and ensure clarity.
